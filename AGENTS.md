# AGENTS.md — plugin-rosetta

Guide for AI coding agents working on this repository.

## Project overview

`plugin-rosetta` is a LinkML-based FHIR-to-OMOP integration library. It:

1. Generates LinkML schemas for OMOP CDM 5.4 from authoritative OHDSI CSV sources
2. Provides hand-authored LinkML schemas for 8 FHIR R4 resources (those mapped to OMOP)
3. Enriches schemas with FHIR→OMOP `exact_mappings` derived from HL7 FHIR-OMOP IG FML files
4. Provides a Python translation layer (FHIR→OMOP) accepting `dict`, `ndjson`, and `Parquet` input
5. Uses Pydantic v2 (generated by LinkML's `gen-pydantic`) + nyctea (Polars+Pydantic wrapper) for validation

## Environment setup

### Prerequisites

- Python >= 3.14
- [`uv`](https://docs.astral.sh/uv/) (package manager)
- [`just`](https://just.systems/) task runner — install with `uv tool install rust-just`

### First-time setup

```bash
uv sync --group dev
just install-nyctea    # critical — see nyctea note below
just download-sources  # downloads OHDSI CSVs + HL7 FML files into sources/
just gen-omop-schema   # generates omop_cdm.yaml / omop_vocabulary.yaml / omop_results.yaml
just gen-pydantic      # generates src/plugin_rosetta/datamodel/plugin_rosetta_pydantic.py
```

### Running tests

```bash
uv run --no-sync python -m pytest
```

All `uv run` invocations in this project use `--no-sync` (see nyctea workaround below).

## Critical constraints and known issues

### nyctea workaround (MUST follow)

`nyctea` has an upstream bug: `requires-python = ">=3.14"` in its `pyproject.toml` (should be `>=3.10`). It is therefore **not listed** in `pyproject.toml` dependencies and **must be installed separately**:

```bash
just install-nyctea
# equivalent to:
# uv pip install --no-deps "nyctea @ git+https://github.com/yannick-vinkesteijn/nyctea.git@6b113f17c2d9fd578e56ca8c89555ac9a71f7130"
```

Consequences:
- Always use `uv run --no-sync` to avoid `uv` overwriting the manually installed nyctea
- Do **not** add nyctea to `pyproject.toml` dependencies until the upstream bug is fixed
- The pinned commit is `6b113f17c2d9fd578e56ca8c89555ac9a71f7130`

### gen-pydantic output must redirect stderr

`gen-pydantic` in linkml 1.10.x emits a `DeprecationWarning` to stderr. The justfile recipe redirects stderr to suppress it so the output file is valid Python:

```bash
uv run --no-sync gen-pydantic \
    --extra-fields ignore \
    src/plugin_rosetta/schema/plugin_rosetta.yaml \
    2>/dev/null > src/plugin_rosetta/datamodel/plugin_rosetta_pydantic.py
```

If you omit `2>/dev/null`, the generated file will start with warning text and fail to import.

### gen-pydantic does not support --split

`gen-pydantic` in linkml 1.10.x generates a **single merged file** only — the `--split` flag is not supported. The output path is always `src/plugin_rosetta/datamodel/plugin_rosetta_pydantic.py`.

### Git-ignored generated files

The following are **git-ignored** and must be regenerated:

| Path | How to regenerate |

| `sources/` | `just download-sources` |

## Schema architecture

### Base URI and prefixes

- Schema base URI: `https://w3id.org/plugin-rosetta/`
- OMOP prefix: `omop:` → `https://athena.ohdsi.org/search-terms/terms#`
- FHIR prefix: `fhir:` → `http://hl7.org/fhir/`

### Schema hierarchy

```
plugin_rosetta.yaml          <- root entry-point
├── omop_cdm.yaml            <- 27 OMOP CDM clinical tables (GENERATED from OHDSI CSV)
├── omop_vocabulary.yaml     <- 10 OMOP vocabulary tables (GENERATED)
├── omop_results.yaml        <- 2 OMOP results tables (GENERATED)
└── fhir_resources.yaml      <- 8 FHIR R4 resources + helper datatypes (hand-authored)
```

### FHIR R4 resources included (exactly 8)

`Patient`, `Encounter`, `Condition`, `Observation`, `Procedure`, `MedicationStatement`, `Immunization`, `AllergyIntolerance`

Note: `MedicationRequest` is **not** included — the HL7 FML structure maps use `MedicationStatement`.

### OMOP tables generated

**CDM schema** (`omop_cdm.yaml`): Person, VisitOccurrence, VisitDetail, ConditionOccurrence, DrugExposure, ProcedureOccurrence, DeviceExposure, Measurement, Observation, Death, Note, NoteNlp, Specimen, FactRelationship, EpisodeEvent, Episode, Survey, CDMSource, Metadata, Cost, CareSite, Location, Provider, PayerPlanPeriod, ObservationPeriod, StemTable, ClinicalEvent

**Vocabulary schema** (`omop_vocabulary.yaml`): Concept, ConceptRelationship, ConceptAncestor, Vocabulary, Domain, ConceptClass, ConceptSynonym, DrugStrength, ConceptRecommendedExposure, SourceToConceptMap

**Results schema** (`omop_results.yaml`): Cohort, CohortDefinition

## Translation layer

### FHIR to OMOP mapping table

| FHIR resource | OMOP table | Translator |
|---------------|-----------|-----------|
| Patient | Person | `translators/fhir_to_omop/patient.py` |
| Encounter | VisitOccurrence | `translators/fhir_to_omop/encounter.py` |
| Condition | ConditionOccurrence | `translators/fhir_to_omop/condition.py` |
| Observation (non-measurement) | Observation | `translators/fhir_to_omop/observation.py` |
| Procedure | ProcedureOccurrence | `translators/fhir_to_omop/procedure.py` |
| MedicationStatement | DrugExposure | `translators/fhir_to_omop/medication.py` |
| Immunization | DrugExposure | `translators/fhir_to_omop/immunization.py` |
| AllergyIntolerance | Observation | `translators/fhir_to_omop/allergy.py` |

### ObservationTranslator routing (important)

`ObservationTranslator.translate_record()` returns `{}` (empty dict, skipped) if the FHIR Observation does not have a `category` code in the known non-measurement set:

```python
OBS_CATEGORIES = {"social-history", "imaging", "survey", "exam", "therapy", "activity", "procedure"}
```

FHIR Observations with `category` = `laboratory` or `vital-signs` are **intentionally skipped** — they are intended for a future `MeasurementTranslator` mapping to the OMOP `Measurement` table. Test inputs for `ObservationTranslator` must include one of the above 7 categories.

### Concept ID gap (known limitation)

All `*_concept_id` fields in translator output are set to `0`. Terminology lookup against the OMOP Vocabulary (Athena) is not yet implemented. This is a deliberate placeholder — concept lookup requires a running OMOP vocabulary database or API.

### Input/output formats

| Format | FHIR input | OMOP output |
|--------|-----------|------------|
| `dict` | `translate_dict(record)` | returned as `dict` |
| ndjson | `translate_ndjson(path)` via orjson | — |
| Parquet | `translate_parquet(path)` via Polars + nyctea | write via `parquet_writer.py`; validate-only via `omop_validator.py` |

## Validation architecture (3 layers)

1. **nyctea** — tabular/columnar DataFrame validation for Parquet inputs (`translators/schemas/`)
2. **Pydantic v2** — record-level structural/type validation (generated by `gen-pydantic --extra-fields ignore`)
3. **LinkML `validate()`** — semantic validation of OMOP output against the LinkML schema

## Known bugs fixed (do not reintroduce)

These bugs were discovered and fixed during initial development. Avoid patterns that would reintroduce them.

### YAML block scalar bug in generate_omop_schema.py

`_yaml_str()` must produce double-quoted scalars, never `>` block scalars. Block scalars inside YAML list items (`- >`) break YAML parsing. The fix uses `" ".join(value.split())` normalization.

### Quoted field name "offset" in OMOP CSV

The OMOP CSV has the field name `"offset"` with surrounding double-quotes. The fix calls `.strip('"')` on `slot_name` in `_render_slot()`.

### ifabsent: 'null' bug

LinkML's Pydantic generator rejects `null` as a value for `ifabsent` on `datetime`/`date` range slots. The fix removes `ifabsent` entirely — `required: false` is sufficient.

### FHIR choice-type slot_uri bug

FHIR choice-type notation (e.g. `fhir:Condition.onset[x]`) is not a valid CURIE/URI and causes LinkML to error. All such `slot_uri` values in `fhir_resources.yaml` use concrete names instead (e.g. `fhir:Condition.onsetDateTime`).

## Design decisions

- **No git submodules** — upstream sources are downloaded at build time via `just download-sources` and stored in `sources/` (git-ignored)
- **No linkml-map** — direct Python translation in each translator module, no mapping framework
- **No MkDocs** — Zensical replaces MkDocs; `mkdocs.yml` was deleted; `zensical.toml` is the docs config; deployment to GitHub Pages is handled by `peaceiris/actions-gh-pages` (Zensical has no `gh-deploy` command)
- **MedicationStatement not MedicationRequest** — the HL7 FML structure maps use `MedicationStatement`
- **uv sync --group dev** (not `--dev`) — required for uv >= 0.4
